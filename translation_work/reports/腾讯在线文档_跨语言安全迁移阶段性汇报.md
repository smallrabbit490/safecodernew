# 跨语言安全代码迁移项目阶段性汇报与方法设计

## 0. 文档用途

本文档用于记录当前阶段的工作整体思路、研究动机、已完成结果、实验发现、参考论文和后续计划。写法上尽量保持两点：

1. 可以快速看清楚项目目前做到哪里、为什么这么做、结果说明什么。
2. 后续 Go、JS、C++、Java 四种语言的更多实验结果可以继续追加，不需要重写整体结构。

## 1. 一句话目标

本项目研究从 Python 这一源语言出发，如何将成对的 Secure/Insecure 安全代码样本迁移到 Go、JS、C++、Java 四种目标语言中。

其中：

| 语言类别 | 目标语言 | 设计意义 |
|---|---|---|
| 稀缺语言 | Go、JS | 用来观察安全经验在相对低资源目标语言上的迁移效果 |
| 常用语言 | C++、Java | 用来观察方法在成熟语言生态中的稳定性和可扩展性 |

这里的核心不是普通代码翻译，而是安全差异迁移：

```text
Python Secure   -> 目标语言 Secure：保留安全防护，并通过功能测试
Python Insecure -> 目标语言 Insecure：保留原始错误示范或漏洞行为
```

换句话说，Secure 要安全可用，Insecure 不能被模型顺手修安全，而要忠实保留原始不安全样本的预期失败或漏洞行为。

## 2. 研究动机

### 2.1 普通代码翻译不够

普通跨语言代码翻译通常只要求功能等价，例如：

```text
Python 功能正确 -> Go/C++/Java/JS 功能也正确
```

但安全代码迁移更复杂。我们面对的是成对样本：

1. Secure 版本：包含安全防护，应该通过功能性测试。
2. Insecure 版本：包含错误示范或漏洞行为，应该复现原本的不安全行为。

因此，模型不仅要会翻译代码，还要理解 Secure 和 Insecure 之间到底差在哪里。

### 2.2 本项目真正关注的问题

本项目关注的问题可以概括为：

```text
Python 中 Secure/Insecure 成对代码之间的安全语义差异，
能否迁移到不同资源水平、不同语言生态的目标语言中？
```

这里包含两个层面的对比：

1. 目标语言差异：Go/JS 和 C++/Java 的迁移难度可能不同。
2. 安全轨道差异：Secure 和 Insecure 的验证标准不同，失败原因也不同。

### 2.3 Insecure 为什么也要验证

Insecure 代码不是失败就算对，也不是越安全越好。它的成功标准是：

```text
目标语言 Insecure 版本应当保留原始 Python Insecure 代码的错误示范或漏洞行为。
```

这可以称为 Negative Behavioral Fidelity，即“不安全行为的忠实迁移”。这是本项目区别于普通安全修复任务的重要点。

## 5. 当前验证流程

### 5.1 总流程

```text
Python 原始样本
  -> 生成目标语言 Secure/Insecure 代码
  -> 沙盒或 Docker 运行测试
  -> 记录 Secure 功能测试结果和 Insecure 行为测试结果
  -> 失败样本进入 repair
  -> repair 后重新验证
  -> 成功候选择优合并回主表
```

### 5.2 Secure 验证

Secure 版本主要验证：

1. 代码能否编译或运行。
2. 功能性测试是否通过。
3. 安全防护是否没有被破坏。

### 5.3 Insecure 验证

Insecure 版本主要验证：

1. 代码能否编译或运行到预期位置。
2. 是否保留原始不安全代码的失败、异常或漏洞行为。
3. 是否避免被模型自动改成安全实现。

### 5.4 为什么要用沙盒或 Docker

生成代码是不可信输入。沙盒和 Docker 的作用是隔离执行环境，避免不安全代码直接影响本机环境，也方便控制超时、依赖和运行资源。

## 6. M1-M7 方法矩阵设计

M1-M7 的目的不是直接作为最终论文方法，而是用来比较不同组件是否有用。

| 方法 | 名称 | 核心思路 |
|---|---|---|
| M1 | feedback repair | 最基础的错误反馈修复 |
| M2 | python delta memory | 引入 Python Secure/Insecure 安全差异经验 |
| M3 | failure-typed repair | 根据失败类型给不同修复策略 |
| M4 | adaptive retrieval | 根据当前样本检索更相关的历史经验 |
| M5 | verifier-guided evolution | 用验证器反馈指导经验更新 |
| M6 | skill evolution | 从失败和成功经验中总结可复用 skill |
| M7 | full method | 将主要组件组合起来 |

### 6.1 方法矩阵的意义

M1-M7 帮助我们回答三个问题：

1. 单纯错误反馈修复是否足够。
2. Python 安全差异经验是否能帮助目标语言生成。
3. 自适应检索、自进化 skill 和完整组合方法是否真的带来提升。

## 7. M1-M7 pilot 实验结果

### 7.1 实验设置

```text
train = 30
dev = 10
test = 10
seed = 20260606
```

注意：这是一轮 pilot 实验，主要用于判断方向，不作为最终论文大规模结论。

### 7.2 有效结果

| 方法 | Records | Secure Rate | Insecure Rate | All-Four Rate | 时间秒 |
|---|---:|---:|---:|---:|---:|
| M1_feedback_repair | 10 | 0.70 | 0.60 | 0.20 | 1719.97 |
| M2_python_delta_memory | 10 | 0.80 | 0.70 | 0.20 | 807.05 |
| M4_adaptive_retrieval | 10 | 0.85 | 0.65 | 0.50 | 1011.39 |
| M6_skill_evolution | 10 | 0.80 | 0.70 | 0.50 | 1293.30 |
| M7_full_method | 10 | 0.80 | 0.80 | 0.40 | 1353.22 |

### 7.3 和 test=2 的对比

| 方法 | test=2 All-Four | test=10 All-Four | 变化 |
|---|---:|---:|---|
| M1_feedback_repair | 0.50 | 0.20 | 明显下降 |
| M2_python_delta_memory | 0.50 | 0.20 | 明显下降 |
| M4_adaptive_retrieval | 0.50 | 0.50 | 保持稳定 |
| M6_skill_evolution | 1.00 | 0.50 | 从满分回落 |
| M7_full_method | 1.00 | 0.40 | 从满分回落 |

### 7.4 观察结论

1. test=2 样本太小，会高估方法效果。
2. M4 和 M6 的 All-Four 最稳定，均为 0.50。
3. M7 的 Insecure Rate 最高，为 0.80，说明它更擅长保留不安全行为。
4. M7 的 All-Four 只有 0.40，说明完整组合方法不是简单全局最优。
5. 后续方法不能只靠堆叠模块，需要更细粒度的路由和修复策略。

## 8. 已有实验结果分析

### 8.1 分析目的

这一部分不是新的 API 实验，而是对已有 test=10 结果做进一步统计分析。目的有两个：

1. 看 M4、M6、M7 三种方法是否各有优势。
2. 判断后续是否需要设计更细粒度的方法选择策略。

### 8.2 关键结果

| 分析项 | 结果 | 含义 |
|---|---:|---|
| 最强固定方法 All-Four | 0.50 | 单一方法目前最高到 0.50 |
| 逐条选择最优方法后的 All-Four | 0.70 | 如果每条样本都能选择最适合的方法，上界可以到 0.70 |
| 只按语言/轨道选择方法后的 All-Four | 0.40 | 只按语言或 Secure/Insecure 轨道选择方法不够 |

这里的“逐条选择最优方法”是一个事后统计上界，不是最终可部署方法。它的作用是说明：M4、M6、M7 之间确实存在互补性，后续方法不能只固定使用某一个版本。

### 8.3 Paired Outcome

| 方法 | C++ Pair | Go Pair |
|---|---:|---:|
| M4 | 0.50 | 0.60 |
| M6 | 0.60 | 0.60 |
| M7 | 0.50 | 0.70 |

观察：

1. M7 对 Go 的 Secure/Insecure 成对一致性更好。
2. M6 在 C++ pair 上更稳。
3. 这说明语言差异会影响最优方法选择。

### 8.4 失败类型

当前失败类型包括：

```text
compile_error
cpp_compile_error
go_undefined_symbol
runtime_error
```

这说明失败不是单一原因，不能只用统一修复提示词。更合理的方式是：

```text
先判断失败类型，再限制修复范围。
```

## 9. 为什么 M1-M7 不够作为最终主方法

M1-M7 是有价值的，但更适合作为 baseline 和 ablation。原因是它们和已有工作存在明显重合：

| 方向 | 与 M1-M7 的关系 |
|---|---|
| 代码翻译 + 执行反馈 | 对应基础翻译和错误反馈修复 |
| 自动程序修复 Agent | 对应 M1/M3 的 repair 流程 |
| RAG / Memory | 对应 M2/M4 的经验检索 |
| Prompt / Skill 自进化 | 对应 M6 的 skill evolution |
| 安全代码生成多轮反馈 | 对应 Secure 版本修复 |

因此，如果论文只写：

```text
我们把 Python 翻译到目标语言，失败后修复，再总结经验。
```

创新性不够强。

更合适的论文主线应该是：

```text
跨语言安全差异迁移。
```

## 10. SCT-Agent 方法设计

### 10.1 方法名称

```text
SCT-Agent: Security-Contrastive Transfer Agent
```

### 10.2 一句话定义

SCT-Agent 不是迁移单段代码，而是迁移 Python Secure/Insecure 成对代码之间的安全语义差异，并在 Go、JS、C++、Java 等目标语言中保持这种差异。

### 10.3 总体流程

```text
Python Secure/Insecure Pair
  -> Security Delta IR
  -> Delta-Aware Router
  -> Dual-Track Translation
  -> Paired Differential Oracle
  -> Risk-Budgeted Repair
  -> Failure Atlas / Evidence-Gated Evolution
```

### 10.4 核心组件

| 组件 | 作用 | 后续可拓展方向 |
|---|---|---|
| Security Delta IR | 抽取 Secure/Insecure 的安全差异 | 扩展为 CWE-aware schema |
| Delta-Aware Router | 根据 CWE、语言、失败类型选择策略 | 学习真实 router，替代 oracle 上界 |
| Dual-Track Translation | 同时约束 Secure 和 Insecure 生成 | 扩展到 Go/JS/C++/Java 四语言 |
| Paired Differential Oracle | 成对验证安全差异是否保留 | 加入更多安全测试和行为测试 |
| Risk-Budgeted Repair | 限制修复范围，防止越修越偏 | 针对不同语言设置不同 repair policy |
| Failure Atlas | 存储失败模式和修复经验 | 加入经验晋升、回滚和跨语言复用机制 |

## 11. 自进化机制设计

### 11.1 为什么需要自进化

不同 CWE、不同目标语言、不同失败类型会产生不同问题。单靠固定 prompt 很难覆盖所有情况。因此需要让系统从失败中总结经验，并在后续任务中复用。

### 11.2 经验来源

| 来源 | 内容 |
|---|---|
| 失败日志 | 编译错误、运行错误、测试失败原因 |
| 修复前后代码差异 | 哪些修改带来了通过 |
| Secure/Insecure 对比 | 安全差异是否被保留 |
| 验证结果 | 修复是否真正有效 |

### 11.3 存储结构

建议把经验分为五类：

| 存储项 | 内容 |
|---|---|
| Security Delta IR | 每条样本的 CWE、安全防护点、缺失防护点、预期行为 |
| Failure Atlas | 按 CWE、语言、轨道、失败类型记录失败模式 |
| Repair Policy | 记录某类失败应该怎么修、哪些地方不能改 |
| Paired Outcome | 记录 Secure/Insecure 成对结果是否保持安全差异 |
| Skill Memory | 经过验证后可复用的修复经验 |

### 11.4 经验晋升规则

经验不能由模型总结后直接进入长期记忆，必须经过验证门控：

```text
只有在 dev/test 样本上带来提升，
且没有破坏 Insecure 行为或 Secure 防护的经验，
才能晋升为长期 skill。
```

如果某条经验导致回归，则需要回滚或降级。

方向：安全代码生成中的多轮反馈与反思。

| M7 不是全局最优 | 不能简单堆叠模块 | 设计 Delta-Aware Router |
| 规则版 Security Delta IR 不够细 | 部分 CWE 识别不充分 | 设计 CWE-aware schema |
| Insecure 容易被模型修安全 | 破坏数据集目标 | 使用 Risk-Budgeted Repair 限制修复范围 |

## 15. 下一步计划

### 15.1 短期计划

1. 汇总 JS/Java 的迁移结果，补齐四语言总表。
2. 选取 Go、JS、C++、Java 各自的失败样本，统一统计失败类型。
3. 实现 SCT-Agent-lite，把 Security Delta IR、paired repair action、Failure Atlas 接入真实 repair prompt。
4. 先用 6-8 个典型失败样本做 targeted test，确认方向有效。

### 15.2 中期计划

1. 将 pilot 实验从 test=10 扩展到 test=30。
2. 比较 M4、M6、M7 和 SCT-Agent-lite。
3. 验证稀缺语言 Go/JS 与常用语言 C++/Java 的差异。
4. 检查 Failure Atlas 中的经验是否能跨语言复用。

### 15.3 论文计划

1. 将 M1-M7 写成 baseline 和 ablation。
2. 将 SCT-Agent 写成主方法。
3. 将 Negative Behavioral Fidelity、Paired Differential Oracle、CWE-aware Security Delta IR、Evidence-Gated Failure Atlas 写成核心贡献点。
4. 最终实验需要覆盖 Base/Plus、四种目标语言、多 seed 或更大样本。

## 16. 阶段性重点结论

1. 当前项目已经不只是做代码翻译，而是在探索跨语言安全差异迁移。
2. Go/C++ 当前已有完整本地验证结果，JS/Java 后续需要统一汇总。
3. M1-M7 实验证明简单堆叠模块不是最好方法。
4. 已有实验结果分析显示不同方法之间存在互补性，因此需要更细粒度的方法选择策略。
5. 论文主线建议升级为 SCT-Agent，而不是直接写 M7 full method。
